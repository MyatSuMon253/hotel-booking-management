# Client Test Plan

## Objective
Document the current client-side unit test strategy for the React/Vite application and define how the existing Vitest suite should be executed, interpreted, and maintained.

## Audience
- Developers extending client features
- Reviewers validating frontend regression coverage
- Academic or project documentation readers who need a clear summary of current frontend QA scope

## Current Test Stack
- `Vitest`
- `jsdom`
- `@testing-library/react`
- `@testing-library/jest-dom`
- `@testing-library/user-event`

## Covered Subsystems
- `src/lib`
- `src/schema`
- `src/components/common`

## Covered Scenarios
- Utility behavior:
  `cn()` class merging
  search-param updates
  date formatting
  amount calculation
  membership discount lookup
  day-range calculations
  timezone adjustment
  amount formatting
  exported filter constants
- Schema validation:
  auth schemas
  user schemas
  booking schema
  buffet schemas
  promotion schema
  review schema
  room schema
  membership-tier schema
- Common component behavior:
  `Loader` render behavior
  `NotFound` 404 rendering and home link
  `Pagination` control rendering and navigation callback behavior

## Out Of Scope
- Apollo client integration and GraphQL request flows
- Page-level form submission and mutation side effects
- Admin pages and dashboards
- Route guards and protected navigation flows
- Invoice generation
- Charts and data visualizations
- File uploads
- Stripe or payment confirmation flows

## Environment And Commands
- Runtime: Node.js
- Test environment: `jsdom`
- Required commands:
  `npm test`
  `npm run test:watch`

## Entry Criteria
- Client dependencies are installed
- Vitest config is available in `vite.config.ts`
- Test setup file is available in `src/test/setup.ts`
- The current test files compile under the Vite TypeScript configuration

## Exit Criteria
- `npm test` completes successfully
- All current client test files pass
- No test requires a live backend, browser automation layer, or external API

## Risks And Limitations
- Current coverage is unit-focused and does not validate full page workflows
- Mock-free schema and helper tests give fast signal, but they do not verify networked UI behavior
- Common component tests cover rendering and small interactions only, not broader layout or route integration
- No current client coverage report is generated by command, so coverage analysis remains procedural until formal coverage tooling is added
